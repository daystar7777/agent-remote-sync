## Evidence Rules

- Mark every scenario `PASS`, `FAIL`, `BLOCKED`, or `NOT_RUN`.
- A `BLOCKED` item needs the exact missing host, tool, permission, or physical
  environment.
- Do not edit production code or tests to hide a defect.
- Redact passwords, bearer tokens, token hashes, TLS private keys, public IPs if
  sensitive, and private hostnames.
- Stop all master/slave/daemon/GUI/Docker lab processes before finishing.
- Include exact commands, report paths, hashes, screenshots, and AIMemory files.
- If an issue is found, separate evidence from suggested fixes.

## Gate B: Browser GUI Evidence

Use a local slave or Docker node and open the controller GUI through
`agentremote open`.

Required actions:

- Upload a file into the current remote folder with no remote item selected.
- Upload a folder into the current remote folder.
- Download a file into the current local folder.
- Download a folder into the current local folder.
- Create a folder from the GUI.
- Rename, move, and delete with confirmation.
- Create a long remote and local file list and prove transfer controls remain
  visible.
- Queue concurrent upload and download jobs.
- Cancel one queued/current transfer and prove the other transfer remains valid.
- Start a large enough transfer to capture progress, speed, ETA, and queued
  entries.
- Open dashboard and verify nodes, processes, routes, calls, approvals, and
  worker policy are visible.

Required evidence:

- Screenshot of two-pane file UI.
- Screenshot of transfer plan dialog.
- Screenshot of transfer monitor during active transfer.
- Screenshot of long-list state with controls visible.
- Screenshot of dashboard overview and one detail view.
- Hash comparison for at least one uploaded file and one downloaded file.
- API or DOM secret check showing no password, bearer token, token hash, command
  fingerprint, or private key exposure.

Pass criteria:

- File/folder transfers do not require selecting a destination file.
- Folder create uses the in-app UI and does not open an unintended browser
  popup/window.
- Cancel only affects the selected job.
- Upload/download queue state remains understandable while jobs are running.

## Gate C: Resume And Fault Injection

Use at least a 256 MiB file; use 1 GiB if disk/time allow.

Cases:

- Kill sender/master during upload, restart, resume, final hash match.
- Kill receiver/slave during upload, restart, resume or structured retry, final
  hash match.
- Kill sender/master during download, restart, resume, final hash match.
- Corrupt one `.agentremote_partial` file manually, retry, and verify either a
  final hash match or a structured failure that does not silently accept bad
  data.
- Run upload and download concurrently while polling dashboard transfer state.
- Verify transfer logs rotate or stay bounded.

Required evidence:

- Exact process start/kill/restart commands.
- Pre/post SHA-256 values.
- Partial directory listing before retry and after success/failure.
- Relevant `.agentremote/logs` or session JSON paths.

Pass criteria:

- No silent data corruption.
- No stale partial remains after successful completion.
- Failure states are structured and visible to the master/controller.

Pass criteria:

- Report shows `PASS 24`, `FAIL 0`, `BLOCKED 0`.
- Unicode manifest hashes pass.
- Mirror delete removes stale files.
- Large file hash is verified.
- No unbounded retry loop, rate-limit false positive, or container leak.

## Gate E: Security And Abuse

Required cases:

- Strict whitelist deny for an unlisted node.
- Explicit allow then successful connection.
- Tailscale CIDR allow import through `agentremote policy allow-tailscale`.
- Wrong password rejection.
- Scoped read/handoff token cannot write.
- TLS self-signed fingerprint trust succeeds.
- TLS fingerprint mismatch fails.
- Rate-limit behavior under repeated bad authentication.
- Panic-on-flood behavior if practical in a safe local lab.

Required evidence:

- Commands and redacted outputs.
- Policy files or sanitized policy summaries.
- TLS fingerprint values redacted or shortened.
- Rate-limit/flood log excerpts with secrets removed.

Pass criteria:

- Deny states fail closed.
- Allow states still require authentication.
- Abuse tests do not leave servers running or firewall state modified.

## Gate F: Real Cross-Host And Cross-OS

Required if hosts are available:

- Windows to Linux or Linux to Windows.
- macOS to Windows/Linux or Windows/Linux to macOS.

Datasets:

- Korean filenames.
- Japanese and Chinese filenames.
- NFC/NFD accent variants.
- Long names near OS limits.
- Case collision candidates on case-insensitive filesystems.

Pass criteria:

- No silent overwrite after Unicode normalization.
- Collision errors are clear.
- Pulled filenames remain readable on the destination OS.
- Manifest/hash evidence is included.

If hosts are unavailable, mark `BLOCKED` with exact missing host pair.

## Gate G: Live Agent Handoff Call/Return

Use two real local roots or two hosts. Send both:

- Instruction-only ask/tell.
- File-plus-instruction handoff.

Required cases:

- Worker in dry-run mode records that it would run.
- Worker in allowed execution mode runs an allowlisted command.
- Worker blocked by policy records a safe blocked report.
- Master receives `STATUS_REPORT`.
- Dashboard and `agentremote map/status` show call progress.

Required evidence:

- Local and remote AIMemory handoff files.
- Call record path.
- Worker report path.
- Dashboard screenshot or sanitized API JSON.

Pass criteria:

- Local and remote histories agree on handoff/call IDs.
- External handoffs are clearly marked as external on the receiver side.
- Reports return to the master without leaking secrets.

## Gate H: Daemon/Profile Lifecycle

Required cases:

- Save two daemon profiles for different project roots.
- Start two slave daemons on different ports.
- Start/open two controller GUIs on different ports.
- Dashboard lists the processes and can stop the selected process only.
- `agentremote daemon install` and `uninstall` dry-run output is correct for the
  host OS.
- Confirm no orphan process remains.

Required evidence:

- `agentremote daemon profile list`.
- `agentremote daemon status`.
- Dashboard process screenshot/API JSON.
- Process table before and after stop.

Pass criteria:

- Project-specific process management does not stop the wrong project.
- Dry-run service commands do not expose passwords.
