## Evidence Rules

- Mark every item `PASS`, `FAIL`, `BLOCKED`, or `NOT_RUN`.
- A `FAIL` must include exact reproduction steps, command output, screenshots,
  and the smallest visible error.
- A `BLOCKED` item must name the missing host, tool, permission, or device.
- Redact passwords, bearer tokens, token hashes, private keys, private hostnames,
  and sensitive public IPs.
- Stop all master/slave/share/daemon/controller/Docker processes before
  reporting.
- Store screenshots under `AIMemory/screenshots/` or another explicit path.
- Do not edit product code to hide failures. Capture evidence first.

## Gate RC-1: Browser GUI Proof

Use a real browser session against the current GUI. Use screenshots.

Required workflows:

- upload a file to the currently open remote folder without selecting a remote
  file;
- upload a folder;
- download a file;
- download a folder;
- create a remote folder;
- rename, move, and delete a remote item with confirmation;
- repeat at least one upload/download with Korean and accented filenames;
- verify local and remote free-space panels update.

Required evidence:

- two-pane UI screenshot;
- transfer plan/confirmation screenshot;
- completed upload/download screenshot;
- remote folder creation screenshot;
- hash comparison for one upload and one download.

Pass criteria:

- buttons remain visible;
- no unintended popup/window opens;
- folder destination behavior is obvious;
- files match by hash.

## Gate RC-2: Queue, Cancel, Speed, ETA

Create enough data to keep transfers active.

Required workflows:

- start upload and download concurrently;
- queue multiple entries behind an active transfer;
- cancel one selected queued/running item;
- prove the other transfer continues or completes;
- capture speed and ETA while active.

Required evidence:

- active transfer monitor screenshot;
- queued stack screenshot;
- selected cancel evidence;
- hash comparison for a non-canceled transfer;
- transfer session/log path.

Pass criteria:

- cancel affects only the selected job;
- speed and ETA are visible during transfer;
- canceled jobs are structured as canceled, not silent success.

## Gate RC-3: Dashboard And Process Usability

Run at least two project-scoped processes.

Required workflows:

- start two shares/slaves on different ports and roots;
- open two controller/master GUI contexts if possible;
- verify dashboard shows nodes, routes, calls, transfers, approvals, inbox, and
  processes;
- stop one selected process from dashboard or CLI process control;
- prove the other process remains alive;
- run daemon install/uninstall dry-run.

Required evidence:

- dashboard overview screenshot;
- node/detail screenshot;
- process list before/after stop;
- dry-run service output with secrets redacted.

Pass criteria:

- process control is project-scoped;
- no orphan process remains;
- dashboard answers "what is connected and what is happening now" quickly.

## Gate RC-4: Native Cross-OS Filename Validation

Run if hosts are available. If not, mark the exact missing pair as `BLOCKED`.

Required pairs:

- Windows to Linux or Linux to Windows;
- macOS to Windows/Linux or Windows/Linux to macOS.

Datasets:

- Korean names;
- Japanese/Chinese names;
- NFC/NFD accent variants;
- case-collision candidates;
- long names near OS limits.

Required evidence:

- OS/filesystem inventory;
- source and destination directory listings;
- manifest and SHA-256 results;
- collision/normalization warning text if applicable.

Pass criteria:

- no silent overwrite;
- filenames remain readable;
- normalization collisions produce clear errors.
