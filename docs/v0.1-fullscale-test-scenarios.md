# v0.1 Full-Scale Test Scenarios

This is the full release gate for agent-remote-sync / `agentremote`. It expands
the earlier validation plan into a multi-environment, multi-node, fault-injection
scenario set. Run it before a public GitHub release or tagged RC.

## Goals

- Prove that the simple daily commands are reliable:
  `setup`, `share`, `connect`, `open`, `send`, `sync-project`, `ask`, `map`,
  `status`, `uninstall`.
- Prove that advanced flows are safe:
  GUI transfer, resumable transfer, project sync, handoff/report, worker policy,
  approval mode, topology/routes, daemon profiles, and dashboard controls.
- Prove failure behavior:
  wrong password, scoped token denial, insufficient storage, permission errors,
  interrupted transfer resume, stale/offline nodes, and flood limits.
- Prove release shape:
  clean install from wheel/GitHub, docs match CLI, no secret leakage, known
  limitations are honest.

## Test Rings

Run in rings. Do not proceed to the next ring if a P0/P1 issue appears.

| Ring | Scope | Required Before Release |
|------|-------|-------------------------|
| R0 | Static and unit/integration tests | Yes |
| R1 | Local two-node headless smoke | Yes |
| R2 | Browser GUI and dashboard | Yes |
| R3 | Fault injection and security | Yes |
| R4 | Project sync and handoff/report | Yes |
| R5 | Worker policy and approval mode | Yes |
| R6 | Daemon, topology, and routes | Yes |
| R7 | Cross-OS filename and platform checks | Recommended for RC, required for stable |
| R8 | Large-file and many-file soak | Recommended for RC, required for stable |
| R9 | Packaging, docs, install/uninstall | Yes |

For a command-oriented lab runbook that expands R7 and R8 into concrete
multi-machine steps, use

For repeatable Linux multi-node validation in containers, use
[`docs/docker-fullscale-lab.md`](docker-fullscale-lab.md).
For the stricter release-gate matrix that combines automated, Docker, VM,
real-host, GUI, security, soak, daemon, and packaging evidence, use

## Environments

Minimum release-candidate matrix:

- Windows 10/11, PowerShell, Python 3.10+.
- macOS, zsh, Python 3.10+.
- Linux, bash, Python 3.10+.
- Browser GUI: Chromium/Edge on Windows, Safari or Chromium on macOS, Chromium
  or Firefox on Linux.
- Network:
  - `127.0.0.1` local-only.
  - LAN or VM host-only network.
  - Tailscale route when available.

If only one OS is available, run all non-cross-OS tests and mark cross-OS tests
as `BLOCKED: environment unavailable`.

## Data Sets

Create reusable test roots:

| Dataset | Shape |
|---------|-------|
| DS-small | 5 small text files, 2 folders |
| DS-unicode | NFC/NFD Korean, Japanese, emoji-free accent filenames |
| DS-conflict | same paths with different content on both sides |
| DS-many | 5,000 to 20,000 files, shallow and deep folders |
| DS-large | 1 GiB file for RC, 5+ GiB for stable soak when disk allows |
| DS-empty | empty folders and empty files |
| DS-permission | read-only target folder or denied file handle |
| DS-project | `.git`, `.venv`, `node_modules`, `.agentremote`, `AIMemory`, `src`, `tests` |

Do not intentionally fill a real disk. Use mocked/free-space hooks where
available for insufficient-storage checks. If a real disk pressure test is run,
use a disposable volume.

## Severity

- P0: path escape, data loss, secret leak, unsafe remote execution, destructive
  action without approval, release blocker.
- P1: install broken, common transfer/sync/handoff flow broken, daemon crashes.
- P2: confusing UX, missing status, recoverable failure path, flaky behavior.
- P3: docs wording, visual polish, non-blocking compatibility issue.

## Required Report

Write:

```text
AIMemory/test-results_YYYYMMDD-HHMMSS-fullscale-deepseek.md
```

Every scenario must include:

- Status: `PASS`, `FAIL`, `BLOCKED`, or `NOT_RUN`.
- Environment and command transcript summary.
- Expected vs actual.
- Evidence paths, screenshots, dashboard URL, or API JSON.
- Severity and minimal repro for failures.

## Release Exit Criteria

For a public v0.1 GitHub release:

- R0, R1, R2, R3, R4, R5, R6, and R9 must have no P0/P1 failures.
- Any P2 must have either a fix or an explicit known-limitation note.
- R7 and R8 may be partially blocked for RC, but blockers must be documented.
- No secret exposure findings may remain open.
- No destructive action may happen without explicit approval.
- Fresh wheel install must pass outside the source tree.
- README and release notes must match the current implementation.

## DeepSeek Deliverable

DeepSeek should return:

1. `AIMemory/test-results_YYYYMMDD-HHMMSS-fullscale-deepseek.md`
2. Screenshots for R2 browser/dashboard scenarios when possible.
3. A sorted issue table with P0/P1/P2/P3 classification.
4. Minimal repro commands for every failure.
5. A list of scenarios marked `BLOCKED` or `NOT_RUN` with reasons.

Codex will review the report, fix issues, rerun relevant tests, and then decide
whether the release is ready for commit/tag/push.
