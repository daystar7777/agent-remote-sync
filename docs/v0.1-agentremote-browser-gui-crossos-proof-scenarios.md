# v0.1 AgentRemote Browser GUI and Cross-OS Proof Scenarios

This phase closes the gap left by automated full-scale validation. Backend,
headless, local RFS, and Docker evidence are already strong. This document is
only for evidence that must come from a real browser session or native host pair.

Do not count API-only or CLI-only evidence as GUI evidence.

## G0: Fresh Live Browser Session

Start a fresh daemon/share and controller GUI. Record:

- exact commands;
- daemon/controller ports;
- process IDs;
- browser URL;
- timestamp.

Before interaction, prove:

```powershell
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:<ui-port>/api/bootstrap
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:<ui-port>/api/dashboard
```

Pass criteria:

- both endpoints return HTTP 200;
- browser tab URL matches the live port;
- browser page is reloaded after server startup;
- stale tabs and offline dashboard states are not accepted as success.

## G1: Browser Upload and Download

From the browser UI:

- upload one local file into the currently open remote folder without selecting
  a remote file;
- upload one local folder;
- download one remote file into the currently open local folder;
- download one remote folder;
- create a remote folder through the UI;
- rename and move a disposable item;
- delete only a disposable item;
- repeat at least one upload/download using Korean and accented filenames.

Required evidence:

- screenshot before transfer;
- transfer plan screenshot;
- active transfer screenshot;
- completion screenshot;
- hash comparison for one upload and one download;
- proof that action buttons remain visible with a long list.

## G2: Queue, Cancel, Speed, ETA

From the browser UI:

- start an upload and a download close together;
- queue at least three jobs/items;
- cancel one queued or running operation;
- prove another operation continues or completes;
- capture speed and ETA while a job is active;
- capture the canceled state.

Required evidence:

- screenshot showing stacked transfer rows;
- screenshot showing speed and ETA;
- screenshot or API JSON showing canceled job state;
- hash evidence for a non-canceled completed transfer.

Pass criteria:

- cancel affects only the selected job;
- speed and ETA are visible during active transfer;
- failed/canceled/completed states are visually distinct.

## G3: Dashboard, Inbox, Process Control

From the same live session:

- open dashboard;
- open node detail;
- open inbox;
- show calls/handoffs;
- show approvals if available;
- stop one disposable project-scoped process from dashboard;
- prove another process remains alive if multiple processes are running.

Required evidence:

- `/api/dashboard` JSON with secrets redacted;
- dashboard screenshot;
- inbox screenshot;
- process list before/after stop.

## G4: Native Cross-OS Filename Pair

Run when hosts are available. If unavailable, mark the exact missing pair
`BLOCKED`.

Required pairs:

- Windows to Linux or Linux to Windows;
- macOS to Windows/Linux or Windows/Linux to macOS.

Dataset:

- Korean filenames;
- Japanese/Chinese filenames;
- NFC/NFD accent variants;
- case-collision candidates;
- names near OS path limits.

Pass criteria:

- no silent overwrite;
- readable filenames on destination;
- normalization/case collisions produce warnings or explicit errors;
- manifest and SHA-256 hashes match.

## G5: Optional Heavy Soak

Run when time and disk permit:

```powershell
python tools\run_docker_fullscale.py --fresh --many-count 5000 --large-size-mib 1024 --down-volumes
```

Pass criteria:

- 24/24 PASS;
- 0 FAIL;
- 0 BLOCKED;
- no `rate_limited`;
- no container/network/volume leak.
