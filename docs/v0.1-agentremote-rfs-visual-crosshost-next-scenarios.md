# v0.1 AgentRemote RFS Visual / Cross-Host Next Scenarios

This scenario set narrows the next phase after the automated local and Docker
gates passed:

- `AIMemory/test-results_20260503-183500-agentremote-local-rfs.md`
- `build/docker-fullscale-results-codex-review/results/test-results_20260503-093325-docker-fullscale.md`
- `AIMemory/test-results_20260503-183402-codex-agentremote-rfs-review.md`

Do not repeat the baseline unless a sanity check is needed. The goal is to
collect the evidence that cannot be proven well by unit tests alone: browser UX,
real cross-host behavior, long-running resilience, and daemon lifecycle.

## Evidence Rules

- Mark each item `PASS`, `FAIL`, `BLOCKED`, or `NOT_RUN`.
- For `FAIL`, include exact reproduction steps and the smallest observed error.
- For `BLOCKED`, name the missing host, tool, permission, or physical device.
- Redact passwords, bearer tokens, token hashes, TLS private keys, private host
  names, and sensitive public IPs.
- Stop every master, slave, daemon, GUI, and Docker container before finishing.
- Store screenshots under `AIMemory/screenshots/` or another explicit path and
  reference them from the report.
- Report command output, API snippets, SHA-256 hashes, process tables, and
  AIMemory handoff paths.
- Do not edit production code just to make a test pass. If a bug is found,
  document evidence first, then suggest the fix separately.

## Gate V1: Browser GUI Core Workflows

Start a slave and controller GUI through current `agentremote` commands. Use the
browser UI, not only CLI calls.

Required actions:

- Upload one file into the currently open remote folder with no remote item
  selected.
- Upload one folder into the currently open remote folder.
- Download one file into the currently open local folder.
- Download one folder into the currently open local folder.
- Create a remote folder with the GUI create-folder action.
- Rename, move, and delete a remote item with confirmation.
- Repeat create/upload/download with Korean and accent-containing filenames.

Required evidence:

- Screenshot of the two-pane file UI before transfer.
- Screenshot of transfer plan or confirmation UI.
- Screenshot after each completed upload/download group.
- Hash comparison for one uploaded file and one downloaded file.
- Note whether any browser popup/window was opened unexpectedly.

Pass criteria:

- Destination folder selection is intuitive and does not require selecting a
  target file.
- Folder create stays in the app UI.
- Unicode filenames remain readable.
- Completed files match by hash.

## Gate V2: Queue, Cancel, Speed, ETA

Create enough data to keep at least two transfers visible at once.

Required actions:

- Start one upload and one download concurrently.
- Queue multiple files/folders behind an active transfer.
- Cancel one selected queued or running entry.
- Verify other queued/running entries continue or complete.
- Capture active speed and ETA.

Required evidence:

- Screenshot of the transfer monitor while active.
- Screenshot showing queued entries.
- Screenshot or API evidence after selected cancel.
- Hash comparison for the transfer that was not canceled.
- Any transfer session JSON or log path related to cancel.

Pass criteria:

- Cancel affects only the selected job.
- Speed and ETA are visible while data is moving.
- The queue remains understandable when several entries are stacked.
- A canceled job leaves a structured state, not a silent partial success.

## Gate V3: Long Lists And Dashboard Usability

Generate long local and remote directories.

Required actions:

- Create at least 300 local files and 300 remote files.
- Scroll both panes.
- Confirm upload/download controls remain visible or reachable.
- Open dashboard overview.
- Open one node detail view.
- Open process, route, call, approval, inbox, and transfer sections.

Required evidence:

- Screenshot of long-list file pane with controls visible.
- Dashboard overview screenshot.
- Dashboard node/detail screenshot.
- Sanitized dashboard API JSON for nodes/processes/calls/transfers.

Pass criteria:

- Primary transfer buttons do not disappear below long file lists.
- Dashboard gives a quick answer for connected nodes, process state, latest
  transfers, latest handoffs, and pending approvals.
- No secret appears in dashboard JSON or DOM text.

## Gate X1: Real Cross-Host / Cross-OS

Run when hosts are available. If not, mark exact missing host pair as `BLOCKED`.

Required pairs:

- Windows to Linux or Linux to Windows.
- macOS to Windows/Linux or Windows/Linux to macOS.

Datasets:

- Korean names: `가나다`, `파일 테스트`, `새 폴더`.
- Japanese/Chinese names.
- NFC/NFD accent variants such as composed and decomposed `é`.
- Case collision candidates such as `Readme.txt` and `README.txt`.
- Long names near host filesystem limits.

Required evidence:

- Source and destination OS/filesystem details.
- `agentremote` command transcript.
- Manifest/hash output.
- Directory listing on both sides after transfer.
- Collision or normalization warning text where applicable.

Pass criteria:

- No silent overwrite after normalization.
- Collision handling is explicit and understandable.
- Pulled filenames remain readable on the destination OS.

## Gate R1: Resume And Fault Injection

Use at least 256 MiB. Use 1 GiB when disk/time allow.

Required actions:

- Kill sender/controller during upload, restart, resume, verify final hash.
- Kill receiver/slave during upload, restart, resume or structured retry,
  verify final hash.
- Kill sender/controller during download, restart, resume, verify final hash.
- Corrupt one `.agentremote_partial` file, retry, and verify hash success or a
  structured failure.
- Poll dashboard transfer state during at least one interrupted transfer.

Required evidence:

- Process start/kill/restart commands.
- Partial directory listing before retry and after completion.
- SHA-256 before/after values.
- Transfer log/session paths.

Pass criteria:

- No silent data corruption.
- No stale partial remains after success.
- Failure is visible to the master/controller.

## Gate S1: Heavy Docker Soak

Preferred:

```powershell
python tools\run_docker_fullscale.py --fresh --many-count 5000 --large-size-mib 1024 --down-volumes
```

Fallback:

```powershell
python tools\run_docker_fullscale.py --fresh --many-count 2000 --large-size-mib 256 --down-volumes
```

Pass criteria:

- Report shows `PASS 24`, `FAIL 0`, `BLOCKED 0`.
- Unicode manifest hashes pass.
- Mirror delete removes stale files.
- Large file hash is verified.
- No unbounded retry loop or container leak remains.

## Gate D1: Daemon / Profile Lifecycle

Required actions:

- Save two daemon profiles for different project roots.
- Start two slaves on different ports.
- Start/open two controller GUIs on different ports.
- Stop one selected process from the dashboard.
- Verify the other process remains alive.
- Run OS service install/uninstall dry-run for the host OS.

Required evidence:

- `agentremote daemon profile list`.
- `agentremote daemon status`.
- Process table before and after dashboard stop.
- Dashboard process screenshot/API JSON.
- Dry-run service command output with secrets redacted.

Pass criteria:

- Project-specific process control does not stop the wrong project.
- No orphan process remains after cleanup.
- Dry-run service output does not expose passwords.
