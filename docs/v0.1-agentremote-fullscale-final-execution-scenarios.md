# v0.1 AgentRemote Full-scale Final Execution Scenarios

This matrix is the final user-confidence pass for `agent-remote-sync`. It
extends the live GUI / cross-OS scenarios with install, headless, topology,
security, process-control, and soak coverage.

Do not mark a scenario complete from stale browser tabs, cached DOM, generated
mock evidence, or unit tests alone. When a scenario needs a live product, prove
the live API first.

## S0: Environment Inventory

Record:

- OS and filesystem for each host;
- Python version and package install mode;
- `agentremote --version` or equivalent package metadata;
- `agent-work-mem` availability;
- Git availability;
- Docker availability;
- browser used for GUI evidence;
- free disk space before and after large transfers;
- ports and process IDs used by each daemon/controller.

Pass criteria:

- missing prerequisites are reported as `BLOCKED`, not silently skipped;
- logs include exact commands and redacted secrets.

## S1: Install, Bootstrap, and Remove

Test from a clean environment:

- install from the repository;
- confirm `agent-work-mem` is required and the user-facing message is clear;
- run bootstrap checks;
- start and stop a daemon/controller;
- remove or uninstall cleanly where supported;
- reinstall and prove previous stale processes do not confuse startup.

Pass criteria:

- common agent commands are short and understandable;
- failure text tells the user what to do next;
- no orphan process remains after cleanup.

## S2: Fresh Live GUI Guard

Start a fresh local daemon/share and controller GUI. Before screenshots:

```powershell
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:<ui-port>/api/bootstrap
Invoke-WebRequest -UseBasicParsing http://127.0.0.1:<ui-port>/api/dashboard
```

Pass criteria:

- both endpoints return HTTP 200;
- browser URL matches the live UI port;
- the page was reloaded after the server started;
- stale/offline pages show a clear restart/reload message and are not counted as
  successful GUI evidence.

## S3: GUI Transfer Workflows

Run these from the browser UI:

- upload a single file into the currently open remote folder without selecting a
  remote item;
- upload a folder;
- download a single file;
- download a folder;
- create a remote folder;
- rename and move a disposable item;
- delete only a disposable item;
- repeat transfer with Korean, Japanese/Chinese, accented, space-containing, and
  long filenames;
- try a transfer larger than remaining destination free space and capture the
  warning or failure report.

Required evidence:

- screenshots before, during, and after transfers;
- hash comparison for at least one upload and one download;
- free-space display on both local and remote sides;
- no unintended popup/window on folder creation;
- transfer buttons remain visible with long file lists.

## S4: Queue, Cancel, Speed, ETA

Exercise concurrent and stacked transfers:

- start upload and download at the same time;
- queue at least three transfer entries;
- cancel one queued job;
- cancel one running job if practical;
- prove remaining jobs continue or complete;
- capture speed and ETA while a transfer is active;
- include folder creation as an operation entry if exposed by the UI/event log.

Pass criteria:

- cancel affects only the selected job;
- canceled jobs have structured state;
- completed jobs have hash evidence;
- failed jobs include actionable error text.

## S5: Headless Transfer and Handoff

Run agent-driven workflows:

- send file only;
- send folder only;
- send instruction only;
- send file/folder plus instruction;
- receive a remote handoff result;
- verify local and remote AIMemory records;
- verify host alias usage;
- test first-use authentication and subsequent reuse;
- test local approval and remote approval request separation.

Pass criteria:

- master/controller receives visible completion or failure status;
- remote-origin handoffs are clearly marked;
- silent mode does not hide exceptional errors;
- permission requests can be approved and reach the waiting agent.

## S6: Dashboard, Topology, and Process Control

Run with multiple project-scoped processes:

- at least two daemons on one machine if ports permit;
- at least one controller GUI;
- at least one headless controller action;
- dashboard overview;
- node detail;
- route or ping status;
- last file transfer summary;
- last handoff call/return summary;
- stop one selected disposable process and prove another stays alive.

Pass criteria:

- dashboard quickly answers what is connected, what is running, and what moved
  recently;
- topology is understandable without reading logs;
- process stop is scoped and reversible by restarting.

## S7: Security and Policy

Validate:

- wrong password/token rejection;
- whitelist allow/deny;
- Tailscale IP import when Tailscale is available;
- route preference and timeout behavior;
- rate-limit/flood simulation;
- TLS or HTTPS/WSS mode if configured;
- secret redaction in logs, screenshots, and reports.

Pass criteria:

- denied events are logged without exposing secrets;
- flood/rate-limited requests do not starve normal local control;
- severe overload fails closed or shuts down cleanly rather than corrupting state.

## S8: Native Cross-OS Filename Validation

Run when host pairs are available. If not, mark exact pairs `BLOCKED`.

Pairs:

- Windows to Linux or Linux to Windows;
- macOS to Windows/Linux or Windows/Linux to macOS.

Datasets:

- Korean filenames;
- Japanese/Chinese filenames;
- NFC/NFD accent variants;
- case-collision candidates;
- long names near OS limits.

Pass criteria:

- no silent overwrite;
- filenames remain readable;
- normalization collisions produce clear warnings or errors;
- manifest and SHA-256 hashes match.

## S9: Docker Full-scale and Soak

Minimum:

```powershell
python tools\run_docker_fullscale.py --fresh --many-count 2000 --large-size-mib 256 --down-volumes
```

Preferred:

```powershell
python tools\run_docker_fullscale.py --fresh --many-count 5000 --large-size-mib 1024 --down-volumes
```

Pass criteria:

- result report is `PASS 24`, `FAIL 0`, `BLOCKED 0`;
- no `rate_limited`;
- Unicode manifest hashes pass;
- large-file hash is verified;
- no container, network, or volume leak remains after cleanup.
