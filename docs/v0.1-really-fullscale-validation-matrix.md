# v0.1 Really Full-Scale Validation Matrix

This matrix is the release-confidence gate after the automated unit, smoke, and
small Docker full-scale runs are green. Every scenario must produce evidence:
commands, screenshots, hashes, logs, AIMemory paths, or Docker report files.

## Evidence Rules

- Mark each item as `PASS`, `FAIL`, `BLOCKED`, or `NOT_RUN`.
- `BLOCKED` needs an exact missing host/tool/environment.
- Do not count `BLOCKED` as `PASS`.
- Do not edit tests to hide a defect.
- Redact passwords, bearer tokens, token hashes, and private host details.
- Stop any started master, slave, daemon, GUI, or Docker lab before finishing.

## RFS-0 Automated Baseline

Commands:

```powershell
python -m compileall -q src tests deepseek-test smoke.py
python -m pytest -q
python smoke.py
python -m pytest tests\test_docker_fullscale_tools.py -q
```

Pass criteria:

- 0 failures.
- Smoke reports all checks passed.
- No generated cache/log artifact is required for git.

## RFS-1 Docker Multi-Node Gate

Commands:

```powershell
python tools\run_docker_fullscale.py --fresh --many-count 50 --large-size-mib 1 --down
python tools\run_docker_fullscale.py --fresh --many-count 500 --large-size-mib 16 --down
```

Optional soak:

```powershell
python tools\run_docker_fullscale.py --fresh --many-count 5000 --large-size-mib 1024 --down-volumes
```

Pass criteria:

- Each report shows `PASS 24`, `FAIL 0`, `BLOCKED 0`.
- Wrong password rejection passes.
- Scoped-token write denial passes.
- Unicode manifest hashes pass.
- Mirror delete removes stale remote files.
- Large-file upload and pull hash evidence is present.

## RFS-2 Browser GUI Gate

Required actions through the browser UI:

- Upload a file into the current remote folder with no remote item selected.
- Upload a folder.
- Download a file into the current local folder.
- Download a folder.
- Create folder.
- Rename, move, and delete with confirmation.
- Queue concurrent upload and download.
- Cancel one queued/current job and verify the other job remains valid.
- Open dashboard and verify nodes, processes, routes, calls, approvals, and
  worker policy are visible.

Required evidence:

- Screenshot of the two-pane file UI.
- Screenshot of the transfer plan dialog.
- Screenshot of transfer monitor with status, speed, and ETA if the file is
  large enough to observe them.
- Screenshot of the dashboard.
- Hash comparison for transferred files.
- DOM or API secret check showing no password/token exposure.

Pass criteria:

- Upload/download controls stay visible with long file lists.
- File/folder transfer succeeds without forcing a destination item selection.
- New folder uses the in-app dialog, not a browser popup/new window.
- Cancel affects only the selected transfer job.

## RFS-3 Resume And Fault Gate

Use at least 256 MiB, 1 GiB when practical.

Cases:

- Kill sender during upload, restart, resume, final hash match.
- Kill receiver during upload, restart, resume or structured retry, final hash
  match.
- Kill sender during download, restart, resume, final hash match.
- Corrupt one partial file, retry, final hash match or structured failure.
- Run upload and download concurrently while polling dashboard transfer state.

Pass criteria:

- No stale `.agentremote_partial` entry after successful completion.
- Corrupt partials do not silently produce a bad destination file.
- Transfer logs rotate or remain bounded.

## RFS-4 Cross-OS Filename Gate

Required real or VM pairs:

- Windows to Linux or Linux to Windows.
- macOS to Windows/Linux or Windows/Linux to macOS.

Datasets:

- Korean filenames.
- Japanese and Chinese filenames.
- NFC and NFD accent variants.
- Long names near OS limits.
- Case-collision candidates on case-insensitive filesystems.
- Mixed path separators from user input.

Pass criteria:

- No silent overwrite after Unicode normalization.
- Collision reports are clear.
- Pulled filenames remain readable on the target OS.
- Manifest/hash evidence is included.

## RFS-5 Agent Handoff Call/Return Gate

Cases:

- `tell` instruction only.
- `handoff` file plus task.
- `ask --wait-report` with worker callback.
- Worker allowed command.
- Worker blocked command.
- Worker timeout.
- Local approval request and decision.
- Remote approval request and decision.

Pass criteria:

- Local and remote AIMemory both identify external handoff context.
- Dashboard or CLI shows states: sent, claimed/running, reported, failed, or
  timeout.
- Approval decision reaches the waiting execution path.
- Auto-run does not execute outside worker policy.

## RFS-6 Daemon, Lifecycle, And Uninstall Gate

Cases:

- Multiple project daemons on one host.
- Multiple GUI controllers on one host.
- Saved daemon profile list/remove.
- Dashboard stop and forget.
- Stale process detection after forced stop.
- Service install dry-run on Windows, macOS, and Linux.
- Reboot persistence on at least one real host if available.
- Uninstall dry-run and project-state cleanup plan.

Pass criteria:

- Profiles store no password or bearer token.
- Dashboard shows process ownership clearly enough to stop the right process.
- Uninstall preserves AIMemory by default.

## RFS-7 Security And Abuse Gate

Cases:

- Whitelist strict deny for an unlisted node.
- Whitelist allow and deny entries by node alias.
- Tailscale CIDR allow import.
- Rate-limit behavior under repeated bad auth.
- Panic-on-flood behavior if enabled.
- TLS self-signed fingerprint trust and mismatch rejection.

Pass criteria:

- Bad authentication does not create connection state.
- Flood response is bounded and logged.
- TLS mismatch does not silently fall back to insecure HTTP.

## RFS-8 Report Template

Write:

```text
AIMemory/test-results_YYYYMMDD-HHMMSS-really-fullscale-deepseek.md
```

Include:

- Environment inventory.
- Exact commands.
- PASS/FAIL/BLOCKED/NOT_RUN matrix.
- Docker report paths.
- Screenshot paths.
- File hashes.
- AIMemory handoff/call/report paths.
- P0/P1 issue list.
- Suggested fixes, but keep fixes separate from test evidence.
